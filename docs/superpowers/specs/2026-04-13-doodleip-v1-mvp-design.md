# DoodleIP v1 MVP — Design Spec

**Date:** 2026-04-13
**Status:** Approved
**Owner:** Weston

## Context and motivation

DoodleIP's landing page (live at `doodleip.vercel.app`) pitches "a doodle IP generator for creators who don't want to show their face but want to be recognizable on Xiaohongshu". The product beyond the landing page was not built.

Three validation cycles in April 2026 resolved what was previously unknown:

1. **Aesthetic feasibility (Step 0)** — Gemini 2.5 Flash Image with a doodle-strategy prompt produces usable hand-drawn IP candidates 4/4 across 8 archetypes and 4 color variants. Over 45 generations, zero AI slop. Effective knobs are negative constraints (`no color`, `no background`, `no shading`) plus imperfection keywords (`deliberately imperfect`, `wobbly`).
2. **Pose variation (Step 1 spike)** — Given an IP reference image, Gemini produces the same character in different poses (thinking / waving / pointing / holding phone) with 4/4 identity preservation.
3. **Scene variation (Step 1 expanded spike)** — Given an IP reference, Gemini produces the character in specific content-driven scenes (at a Tesco sign, in a tube station, holding a SIM card) with 4/4 identity preservation AND automatic emotional expression matching (worried eyebrows at Tesco, sweat drop at the tube station).

Market research surfaced a second load-bearing insight: on Xiaohongshu, the IP character contributes ~20% of creator success; content and 文案 carry 80%. Successful doodle IPs on XHS are not "cute characters with nice posts" — they are "creators with strong content whose recurring character makes them recognizable".

These findings together justify a meaningfully different v1 than the original `DESIGN.md` envisioned. The v1 scope is a single-user product that turns pasted text into a publishable Xiaohongshu carousel where the user's IP acts out the content scene-by-scene.

## Goal

Ship a working product that lets a Xiaohongshu creator create their own hand-drawn IP character once, then repeatedly turn pasted text into a 5-7 page carousel where the IP appears in a different content-specific scene on each page — publishable to Xiaohongshu by manual upload.

## Target users

One user archetype in v1: a Xiaohongshu content creator who

- writes their own content (text posts, articles, notes), and
- wants a recurring visual identity without showing their face, and
- cannot or will not draw it themselves.

Light vertical signal (留学 / AI知识 / 打工人 / 读书 / 通用) is captured during onboarding to tune IP generation but does not branch product flow.

## Core decisions

The 11 design decisions aligned during brainstorming:

| # | Decision | What it means |
|---|---|---|
| 1 | Job-to-be-done is "A" | User has written content → product produces a ready-to-publish carousel. Not a content-generator for users with only a topic idea. |
| 2 | IP architecture is scene-variation | Each body page is an AI-generated IP-in-scene image, not a static IP sticker pasted on text pages. IP acts out the content. |
| 3 | Carousel composition is LLM-plans-code-composites | Claude Sonnet plans the carousel (page count, per-page scene description, per-page overlay text). Gemini generates each scene. Client-side Canvas renders the text overlay on top. Text is never generated by the image model. |
| 4 | Identity is email magic link | Supabase Auth magic link. No passwords. Single IP per user. Cross-device session via email login. |
| 5 | Vertical presets are light | Onboarding asks "what do you post?" — this text only seeds the IP-generation prompt. No branching in content planning, no custom templates per vertical. |
| 6 | Onboarding is B+D hybrid | Inspiration grid of 8 pre-made archetypes → user picks "vibe" → feature chip selection → optional free text → 4 candidates generated → user picks one → background job generates pose library → user names IP. ~90 seconds, ~$0.50 API cost. |
| 7 | Output is Gallery with per-page regen + ZIP | 5-7 page preview. Each page has a "regenerate" button (~$0.05). Single "download all" button produces a ZIP the user uploads to Xiaohongshu manually. |
| 8 | Pricing is fully free for v1 | No paywall, no subscription. v1 validates whether people repeatedly use it, not whether they pay. ~$160/month API cost at 100 users × 1 post/week is acceptable for Weston. |
| 9 | LLM is Claude Sonnet 4.5, image model is Gemini 2.5 Flash Image | Claude for planning quality. Gemini for image generation (already validated). |
| 10 | Canvas composition runs client-side | User's browser composites scene image + text overlay. Zero backend compute cost. JSZip bundles final ZIP client-side. |
| 11 | Tech stack extends the existing repo | Next.js 16 + React 19 + Tailwind 4 (existing). Supabase Auth + Postgres + Storage (existing, plus new tables). Vercel deploy (existing). |

## Architecture

### Data flow — daily use

```
User pastes text into textarea
        │
        ▼
POST /api/carousel/plan  (Claude Sonnet)
  input  : { userText, userId }
  output : {
    pages: [
      { sceneDescription, overlayText },
      ... 5-7 pages total
    ]
  }
        │
        ▼
For each page, in parallel (Promise.all):
  POST /api/carousel/page  (Gemini 2.5 Flash Image with reference image)
    input  : { userId, sceneDescription }
    output : scene-image.png (1024×1024, server uploads to Supabase Storage,
                              returns signed URL)
        │
        ▼
Client composites overlayText on top of each scene image
  (Canvas API, native, zero dependency)
        │
        ▼
Gallery renders 5-7 pages with per-page regenerate button
        │
        ├── User clicks "regenerate page N" → calls /api/carousel/page again
        │     with a fresh prompt variant → replaces that page in the gallery
        │
        └── User clicks "Download all" → client bundles all composited PNGs
              into a ZIP via JSZip → browser download
```

### Data flow — onboarding

```
User opens /app
        │
        ▼
Supabase Auth check:
  - No session → show email input → send magic link
  - Returning user with IP → redirect /app/generate (skip onboarding)
  - Returning user without IP → redirect /app/onboarding
        │
        ▼
Onboarding step 1 (inspiration): show 8 archetype thumbnails
  Assets: pre-generated (stored as static files in /public or Storage)
  User picks one → archetype_seed stored in client state
        │
        ▼
Onboarding step 2 (features): chip selection + optional free text
  Chips: [戴眼镜, 围巾, 耳机, 马尾, 背包, ...] (static list, 10-15 options)
  Optional description field for creative users
        │
        ▼
POST /api/ip/create  (Gemini, 4 parallel calls)
  input  : { archetype_seed, feature_chips, description, vertical_hint }
  output : 4 candidate images → Supabase Storage → 4 signed URLs
        │
        ▼
Onboarding step 3 (pick): user picks 1 of 4 (or "regenerate all" to retry)
        │
        ▼
POST /api/ip/confirm
  input  : { chosen_candidate_url }
  action : persist IP record to DB; kick off pose library generation
           (6-8 parallel Gemini calls with the chosen image as reference)
  output : { ip_id, pose_library_progress }
        │
        ▼
Onboarding step 4 (name, while pose library generates in background):
  Input: "Give your IP a name (optional)"
  After pose library completes → show "Meet your IP" gallery (6-8 poses) as
  an onboarding reward moment → "Start creating" button → /app/generate

Purpose of the pose library in v1: an onboarding-reward showcase ("here's
your IP in 8 situations") that gives the user a tangible sense of owning
a real character. Daily carousel generation does NOT use pose library —
it generates scenes fresh from the reference image. The library costs
~$0.32 per onboarding; this is an accepted one-time cost for the
product-moment value.
```

### Data model (Supabase Postgres)

```sql
-- Waitlist (existing, unchanged)
-- waitlist (id, email, created_at)

-- IPs — one per user in v1
create table ips (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references auth.users on delete cascade unique,
  name text,
  archetype_seed text,                    -- one of the 8 predefined seeds
  creation_prompt text,                   -- the assembled Gemini prompt
  reference_image_path text not null,     -- Storage path to the picked candidate
  pose_library jsonb default '[]',        -- [{ key: "thinking", path: "..." }, ...]
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

-- Posts — many per user
create table posts (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references auth.users on delete cascade,
  user_text text not null,
  plan jsonb not null,                    -- LLM plan output
  page_image_paths jsonb default '[]',    -- [{ index: 0, path: "..." }, ...]
  created_at timestamptz default now()
);

-- Per-page regen audit (useful for quality learning)
create table page_regenerations (
  id uuid primary key default gen_random_uuid(),
  post_id uuid references posts on delete cascade,
  page_index int not null,
  new_image_path text not null,
  created_at timestamptz default now()
);

-- RLS on all three: user_id = auth.uid()
```

### Supabase Storage (private buckets)

```
ip-references/{user_id}/ip.png                 ← the picked IP candidate
ip-poses/{user_id}/{pose_key}.png              ← 6-8 poses
post-scenes/{post_id}/page-{n}.png             ← each generated scene
```

All access via short-lived signed URLs requested from the authenticated user's session.

## In scope for v1

- Email magic link auth via Supabase Auth
- Onboarding flow: inspiration grid → chip features → 4 candidates → pick → pose library → name
- Single IP per user, stored with reference image + pose library
- Daily flow: paste text → LLM plan → parallel scene generation → Canvas text composite → gallery with per-page regen → ZIP download
- Light vertical signal during onboarding ("what kind of content do you make?") that only adjusts the IP generation prompt
- Chinese-language UI throughout
- Pre-generated asset library: 8 archetype thumbnail images for the inspiration grid (one-time asset creation, checked into `/public` or similar)
- Client-side Canvas compositor for text-on-scene rendering
- Client-side JSZip bundling for download

## Out of scope for v1 (explicit deferrals)

| Deferred feature | Rationale |
|---|---|
| Multiple IPs per user / IP switching | Single recurring IP is the point; multi-IP dilutes identity |
| IP editing after creation | Requires running onboarding again; v1.1 |
| Post history UI / saved workspace | Data is in DB but no UI surface; v1.1 |
| Editing text overlays in gallery | Users have fixed content they want to post |
| Reordering or deleting carousel pages | LLM's page plan is final |
| Direct publish to Xiaohongshu | No public API; user uploads manually |
| Multi-language UI (English, etc.) | Chinese-only v1 |
| Analytics beyond server logs | Deferred until we have users |
| Admin dashboard | N/A at v1 scale |
| Paid tier, subscription, usage limits | v1 is free |
| Content calendar / multi-post planning | This is Job-to-be-done "C" which we deferred |
| IP animation or video output | Image-only v1 |
| Custom font or color palette selection | Fixed crayon aesthetic + default fonts |
| Team / organization accounts | Single-user v1 |
| Rate limiting beyond Supabase and API-provider defaults | Not critical at v1 scale |
| Vertical-specific carousel recipes (留学 gets diary-style 7-page structure, AI gets info-card 5-page structure) | User chose "light" preset depth; vertical tuning is distribution work, not product work |
| "Shareable workspace URL" feature from anonymous-UUID version | Email login replaced the UUID identity mechanism |
| Pose library regeneration if user is unhappy with initial poses | Deferred; users can delete and redo onboarding |

## Success criteria

### Technical thresholds (minimum ship bar)

- Onboarding completion rate (user reaches "done" state) ≥ 80%
- Single-page generation success rate (recognizable IP + well-fitting scene) ≥ 90% on first try
- End-to-end latency: onboarding ≤ 90s, daily use (paste to gallery) ≤ 60s
- Per-page regeneration ≤ 10s
- Zero garbled characters in output text (guaranteed by code-side rendering)
- Generated ZIPs open in standard tools and contain 5-7 PNG images

### Product success signals (observed in the first 2-4 weeks after launch)

| Signal | Threshold | Interpretation |
|---|---|---|
| 🟢 Non-Weston user completes onboarding | ≥ 1 | Product functionally works for strangers |
| 🟢 User returns within 7 days to generate 2nd post | ≥ 3 distinct users | Stickiness signal present |
| 🟢 Generated post published to Xiaohongshu | ≥ 1 confirmed instance | Full loop closes |
| 🟡 Per-post regen rate | 30-50% | AI variance is real but acceptable |
| 🔴 Per-post regen rate | > 50% | AI quality insufficient — invest in prompt engineering |
| 🔴 Weeks 1-2: < 10 users complete onboarding | — | Distribution problem, not product problem |
| 🔴 Among onboarded users: < 20% generate a 2nd post | — | Product is a novelty, not a repeat tool |

### Learning outputs (v1 must produce data answering these)

- Which verticals are over-represented among early users (informs v1.1 preset depth decisions)
- Which of the 8 archetype seeds is picked most often (market aesthetic signal)
- Which feature chips are used most (individuation preferences)
- Session duration during daily use (ideally < 5 min; > 5 min signals friction)
- Which post types (subject matter, text length, tone) produce the highest vs lowest image quality (informs LLM prompt iteration)

## Risks and open questions

### Known risks

- **Gemini rate limits** — v1 expects concurrent image generations (4 for onboarding candidates, 5-7 per post). If Gemini's rate ceilings are hit under load, user-facing failures become visible. Mitigation: exponential backoff retry with one automatic retry; surface explicit "try again in a moment" UX on failure.
- **Text overlay readability on busy scenes** — Some scene outputs may have visual clutter near where text will be composited. Mitigation: the dark caption strip design chosen in the mockup creates a high-contrast band regardless of the scene underneath.
- **Scene consistency drift across many posts** — Across dozens of posts, subtle character drift may accumulate (each scene is generated fresh from the stored reference image, so drift is bounded by Gemini's reference-preservation quality). Not a v1 blocker (any user only sees their own outputs), but worth monitoring as learning data.
- **Supabase default SMTP may throttle** — Supabase's built-in email is limited to 3/hour. If v1 attracts faster sign-ups, magic links get delayed. Mitigation: switch to Resend (free 100/day) if sign-up rate exceeds 3/hour sustained.

### Open questions (to be resolved during implementation, not brainstorming)

- Exact pose library composition (which 6-8 poses to pre-generate) — to be decided during implementation with Weston's input, probably: standing / thinking / waving / pointing / holding-something / sitting / walking / laughing.
- Exact chip list for onboarding features — to be decided during implementation; initial proposal: [戴眼镜 / 戴帽子 / 围巾 / 耳机 / 马尾 / 短发 / 胡子 / 背包 / 相机 / 水壶].
- Claude prompt for content planning — to be designed and iterated during implementation; structure will be: system prompt defining output JSON schema + user message = user's raw text.
- How to pre-generate the 8 archetype thumbnails for the inspiration grid — probably run the existing `experiments/step0-ip-aesthetics` pipeline once with specific seeds and commit the outputs to `/public`.

## Appendix: What this replaces

The original `DESIGN.md` and the earlier Step-0 plan assumed:
- The MVP would be a one-time content generator (no accounts, no persistence)
- IP would be pasted as a static PNG sticker onto fixed HTML templates
- The product thesis hinged on "character consistency across 6-8 poses"

All three assumptions are revised:
- Accounts + single persistent IP is v1
- Scene variation replaces sticker-on-template
- Pose consistency was validated trivially (4/4 green); the harder constraint now is scene-level identity preservation (also validated, 4/4 green)
